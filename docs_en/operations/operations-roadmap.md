## 1. Development Sequence and Dependencies (Development Sequence Roadmap)

Source: `development_sequence_roadmap.md`

### 1.1 Goal

Convert all stabilization backlogs from "problem lists" to "development sequences", answering:

- If starting implementation now, which batch to do first
- What can be done in parallel
- What must wait for prerequisites to complete
- How far each batch needs to go before entering the next batch

### 1.2 Usage Principles

- High priority does not mean it can be done immediately; must follow dependency order
- First do the foundation that can reduce system instability sources, then do capability enhancements
- If a batch's exit criteria are not met, should not proceed to the next batch
- Before Stable Core is stable, do not expand to remote workers, marketplace, multi-tenancy, or complex evolution

### 1.3 Overall Rhythm

```
Week 1:  State / Event / Workflow
Week 2:  Runtime / Tool / Cancel
Week 3:  DB / Recovery / Reconcile
Week 4:  Observability / Ops / Takeover
Week 5:  Quality Gates / Golden Tasks / Rollback
Phase 1b Batch 1: Orchestration / Task Board
Phase 1b Batch 2: Compaction / Edit
Phase 1b Batch 3: Replay / Backpressure
Week 6+:  Remote / PG-Redis / Enterprise Prep
```

### 1.4 Batch Status Enumeration

| Status | Meaning |
| --- | --- |
| `not_started` | Not yet entered this phase or batch |
| `ready` | Passed gate, can start, but not yet started |
| `in_progress` | Started, actively pushing forward |
| `blocked` | Started but blocked |
| `done` | Reached current phase acceptance criteria |

## 2. Architecture Upgrade Roadmap

Source: `architecture_upgrade_roadmap.md`

### 2.1 Goal

Define the top 4 architecture upgrade lines when pushing from the current Phase 1a baseline to the final platform goal.

### 2.2 Current Priority

1. `runtime → execution plane`
2. `transaction storage → data plane`
3. `approval/sandbox/budget → governance control plane`
4. `billing/tenant → tenant and monetization plane`

### 2.3 Overall Principles

- First clarify the platform layer design, then enter corresponding implementation
- Each upgrade line must have contract, phase goals, and exit criteria
- Long-term platform implementation is not allowed without upper-level contract

## 3. Industrial Production Readiness Roadmap

Source: `industrial_production_readiness_roadmap.md`

### 3.1 Goal

Define the path from "runnable framework" to "industrial-grade production system".

Supplementary notes:

- This section describes the industrial-grade target state roadmap, not the current phase1-4 authoritative release level.
- Items involving `blue-green / gray-scale / canary / staged / auto rollback` should be treated as industrial-grade or `M2` expansion goals in the current repository scope, and should not be inversely interpreted as current `off / suggest / shadow` release levels being completed.

### 3.2 Core Principles

- First fill reliability, operations, security, rollback, and manual takeover
- Do not substitute continued business feature expansion for production backing capabilities
- Any "industrial-grade" claim must be supported by contract, runbook, alerting, and rollback path

### 3.3 P0 Roadmap (Production Blockers)

1. Task lease + fencing token
2. PostgreSQL / Redis production roadmap
3. Distributed locks
4. Idempotency and side effects system
5. SLO / Alerting / Runbook
6. Blue-green / gray-scale / rollback
7. Enterprise secret management
8. Audit chain and retention policy
9. Admin control plane and manual takeover
10. Prompt / Model / Policy governance
11. LLM suggestions, code decision boundaries

### 3.4 P1 Roadmap (Enterprise Governance and Isolation Enhancement)

- Multi-tenant isolation enhancement
- Data classification and grading
- Compliance evidence chain
- Resource pool and tenant quota isolation
- On-call and handover system
- Environment layering and configuration center governance
- Architecture governance and schema version governance
- Supply chain and dependency security
- trace / RCA / business and technology dual dashboard
- Workflow static analysis and compensation closure

### 3.5 P2 Roadmap (Scale and High Availability Enhancement)

- HA coordinator
- Hot upgrade and lossless migration
- Anomaly detection
- Auto damage control
- Cross-region deployment
- Remote coordination and disaster recovery
- Memory quality and decay governance
- License / capability engineering tiers
- More mature HITL experience and explainability

### 3.6 Roadmap

```
Phase 1a/1b: Runnable Foundation
        ↓
P0: Production Backing Capabilities
        ↓
P1: Enterprise Governance and Isolation Enhancement
        ↓
P2: Scale and High Availability Enhancement
```

## 4. System Improvement Roadmap

Source: `system_improvement_roadmap.md`

### 4.1 Goal

Organize current system-level improvement suggestions into formal promotion roadmaps to prevent improvements from staying only in chat conclusions.

### 4.2 Execution Principles

- First tighten the 5 foundations: state, error, event, recovery, security
- First clear the most likely sources of instability for online incidents, then consider feature expansion
- First stabilize the capabilities actually needed for the current phase, then enter the next phase
- Any super-phase capability not entered in the formal phase goals is not implemented by default

### 4.3 Improvement Priority

**P0 (Must do first)**:

- State machine and transition audit
- Error code normalization
- Event reliability (Tier-1/Tier-2)
- Recovery and replay integrity

**P1 (High priority support items)**:

- Session / Execution unified model
- Multi-tenant isolation
- Resource limits and budget control

**P2 (Do after running stable)**:

- Complex orchestration and multi-agent
- External provider extension
- Cross-region deployment
