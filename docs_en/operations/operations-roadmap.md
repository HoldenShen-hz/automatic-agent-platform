## 1. Development Sequence and Dependencies (Development Sequence Roadmap)

Source: `development_sequence_roadmap.md`

### 1.1 Goal

Convert all stabilization backlog from "problem list" to "development sequence", answering:

- Which batch to start implementation first
- What can be done in parallel
- What must wait for prerequisites to complete
- What degree each batch must reach before entering the next batch

### 1.2 Usage Principles

- High priority does not equal can be done immediately; must follow dependency order
- First do the base that reduces system instability sources, then do capability enhancements
- If a batch's exit threshold is not met, should not enter the next batch
- Before Stable Core is stable, do not expand to remote worker, marketplace, multi-tenant, or complex evolution

### 1.3 Overall Rhythm

```
Week 1:  State / Event / Workflow
Week 2:  Runtime / Tool / Cancel
Week 3:  DB / Recovery / Reconcile
Week 4:  Observability / Ops / Takeover
Week 5:  Quality Gates / Golden Tasks / Rollback
Phase 1b Batch 1: Orchestration / Task Board
Phase 1b Batch 2: Compaction / Edit
Phase 1b Batch 3: Replay / Backpressure
Week 6+:  Remote / PG-Redis / Enterprise Prep
```

### 1.4 Batch Status Enumeration

| Status | Meaning |
|--------|---------|
| `not_started` | Not yet entered that phase or batch |
| `ready` | Passed gate, can start, but not yet started |
| `in_progress` | Started, actively advancing |
| `blocked` | Started but has blockers |
| `done` | Reached current phase acceptance line |

## 2. Architecture Upgrade Roadmap

Source: `architecture_upgrade_roadmap.md`

### 2.1 Goal

Define the 4 most priority architecture upgrade lines when pushing from current Phase 1a baseline to final platform goal.

### 2.2 Current Priority

1. `runtime → execution plane`
2. `transaction storage → data plane`
3. `approval/sandbox/budget → governance control plane`
4. `billing/tenant → tenant and monetization plane`

### 2.3 Overall Principles

- First clarify the platform layer design, then enter corresponding implementation
- Each upgrade line must have contract, phase goals, and exit thresholds
- Not allowed to directly do long-term platform implementation without upper-level contract

## 3. Industrial Production Readiness Roadmap

Source: `industrial_production_readiness_roadmap.md`

### 3.1 Goal

Define the path from "runnable framework" to "industrial-grade production system".

Supplementary notes:

- This section describes the industrial-grade target state, not the current phase1-4 authoritative release level.
- Items involving `blue-green / gray / canary / staged / auto rollback` should be treated in the current repository metric as industrial-grade or `M2` extended targets, and must not be reversely interpreted as having completed release levels beyond current `off / suggest / shadow`.

### 3.2 Core Principles

- First supplement reliability, operations, security, rollback, and human takeover
- Do not substitute continuing to expand business functions for production guarantees
- Any "industrial-grade" declaration must be supported by contract, runbook, alerts, and rollback path

### 3.3 P0 Roadmap (Production Blockers)

1. Task lease + fencing token
2. PostgreSQL / Redis production roadmap
3. Distributed lock
4. Idempotency and side effects system
5. SLO / Alerts / Runbook
6. Blue-green / Gray / Rollback
7. Enterprise secret management
8. Audit chain and retention policy
9. Admin control plane and human takeover
10. Prompt / Model / Policy governance
11. LLM suggestions, code arbitration boundaries

### 3.4 P1 Roadmap (Enterprise Governance and Isolation Strengthening)

- Multi-tenant isolation strengthening
- Data classification and grading
- Compliance evidence chain
- Resource pool and tenant quota isolation
- On-call and handover system
- Environment layering and configuration center governance
- Architecture governance and schema version governance
- Supply chain and dependency security
- trace / RCA / business + technical dual dashboard
- Workflow static analysis and compensation closed-loop

### 3.5 P2 Roadmap (Scaling and High Availability Enhancement)

- HA coordinator
- Hot upgrade and lossless migration
- Anomaly detection
- Automatic stop-loss
- Cross-region deployment
- Remote coordination and disaster recovery
- Memory quality and decay governance
- License / capability engineering layering
- More mature HITL experience and explainability

### 3.6 Roadmap

```
Phase 1a/1b: Runnable base
        ↓
P0: Production guarantee capabilities
        ↓
P1: Enterprise governance and isolation strengthening
        ↓
P2: Scaling and high availability enhancement
```

## 4. System Improvement Roadmap

Source: `system_improvement_roadmap.md`

### 4.1 Goal

Formalize current system-level improvement suggestions into an advancement roadmap, preventing improvement items from staying only in chat conclusions.

### 4.2 Execution Principles

- First tighten the 5 bases: state, error, event, recovery, security
- First clear the instability sources most likely to cause online incidents, then consider expanding features
- First stabilize the capabilities actually needed for the current phase, then enter the next phase
- Any super-phase capability that has not entered formal phase goals is not implemented by default

### 4.3 Improvement Priority

**P0 (Must do first)**:

- State machine and transition audit
- Error code normalization
- Event reliability (Tier-1/Tier-2)
- Recovery and replay completeness

**P1 (High priority support items)**:

- Session / Execution unified model
- Multi-tenant isolation
- Resource quotas and budget control

**P2 (Do after running stable)**:

- Complex orchestration and multi-Agent
- External Provider extension
- Cross-region deployment
