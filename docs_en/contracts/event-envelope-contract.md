# v4.3 Event Envelope Contract

> v4.3 canonical contract. Covers `EventEnvelope` / `PlatformFactEvent` / `OapeflirViewEvent`.

## 1. Scope

The event system is divided into truth fact and view projection. `platform.*` is the sole truth fact namespace; `oapeflir.view.*` and `oapeflir.rationale.*` only express OAPEFLIR semantic projections, explanations, and audit views.

## 2. EventEnvelope

Minimum fields:

| Field | Type | Description |
| --- | --- | --- |
| `schema_version` | `string` | Envelope wire schema version; used for cross-storage, cross-queue, and cross-language compatibility |
| `eventId` | `string` | Event ID |
| `eventType` | `string` | Event type |
| `eventVersion` | `string` | Event schema version |
| `idempotency_key` | `string` | Producer-side idempotency key; used for append/relay/consumer deduplication |
| `aggregateType` | `string` | Aggregate type |
| `aggregateId` | `string` | Aggregate ID |
| `aggregateSeq` | `number` | Aggregate sequence |
| `tenantId` | `string` | Tenant |
| `traceId` | `string` | Trace |
| `causation_id` | `string?` | Directly upstream event or command ID that triggered this event |
| `correlationId` | `string?` | Correlation chain |
| `partition_key` | `string` | Partition routing key; the same truth aggregate must be stably routed to the same partition strategy |
| `ttl` | `duration?` | Retention/expiration hint; used for relay, cache, view rebuild, or temporary projection expiration strategy |
| `payloadHash` | `string` | Payload hash |
| `payload` | `json` | Payload |
| `occurredAt` | `timestamp` | Occurrence time |

Rules:

- `(aggregateType, aggregateId, aggregateSeq)` must be unique.
- `schema_version`, `idempotency_key`, and `partition_key` are all required; events missing these must not enter the canonical event bus.
- `ttl` only controls envelope lifecycle policy and must not change the audit retention obligations of committed truth facts.
- Tier 1 platform facts must support per-consumer ack and replay.
- Event append must be transactional with truth mutation.

## 3. PlatformFactEvent

`PlatformFactEvent` is a fact event whose `eventType` starts with `platform.`.

Initial namespaces:

- `platform.harness_run.*`
- `platform.plan_graph.*`
- `platform.graph_patch.*`
- `platform.node_run.*`
- `platform.node_attempt.*`
- `platform.side_effect.*`
- `platform.budget.*`
- `platform.decision.*`
- `platform.hitl.*`
- `platform.version_lock.*`

Rules:

- Truth projector, recovery scanner, budget projector, and side-effect projector must only consume `platform.*`.
- Platform facts must not be directly generated by UI view projector; they must come from authoritative paths such as state machine, admission, scheduler, budget, side-effect, and decision.

## 4. OapeflirViewEvent

`OapeflirViewEvent` is a projection event whose `eventType` starts with `oapeflir.view.` or `oapeflir.rationale.`.

Purpose:

- StageRationale
- TraceProjection
- Audit View
- Explain timeline
- Learning/improve/release semantic views

Rules:

- Must not drive `HarnessRun`, `NodeRun`, `BudgetLedger`, `SideEffectRecord` truth mutations.
- Must declare `derivedFromEventId` or `derivedFromEventIds`.
- Can be rebuilt after loss; must not be used as the sole audit fact.

## 5. Legacy/Deprecated Mapping

| Legacy Name | v4.3 Semantic |
| --- | --- |
| `task.*` | Legacy event; after migration should be projected or converted to `platform.harness_run.*` |
| `workflow.*` | Legacy event; new execution facts use `platform.*` |
| `oapeflir.*` | Default projection; only becomes platform fact after explicit adapter conversion |
| `dispatch:*` / `worker:*` | Execution diagnostics or platform facts depend on registry declaration; truth consumers only look at `platform.*` |

## 6. Testing Requirements

- Truth consumers do not consume `oapeflir.view.*`.
- OAPEFLIR view events must carry source facts.
- Platform fact replay can rebuild HarnessRun/NodeRun/Budget/SideEffect read models.
- Production of unregistered event types must fail at startup inspection or testing.

## v4.3 Architecture Remediation

The following items fix contract deviations recorded in `platform-architecture-implementation-consistency-audit.md`. If historical sections of this document conflict with this section, this section, `docs_zh/architecture/00-platform-architecture.md`, ADR-109 through ADR-113, and `src/platform/contracts/executable-contracts/` take precedence.

- T-5: Missing the 5 required fields required by architecture ContractEnvelope: schema_version/idempotency_key/causation_id/partition_key/ttl. Root cause: Early documents only described event fact storage fields, omitting envelope-layer idempotency, partitioning, and lifecycle metadata. Fix: These 5 fields have now entered the `EventEnvelope` canonical minimum fields; old camelCase/omitted-field conventions can only be used as adapter or migration inputs, not as new implementation entry points.

Mandatory rules: State transitions must go through `RuntimeStateMachine.transition(command)`; execution plans must use `PlanGraphBundle`; execution results must use `NodeAttemptReceipt`; truth events must only use `platform.*`; OAPEFLIR can only be `oapeflir.view.*`/rationale projections; budgets must use `BudgetLedger`/`BudgetReservation`/`BudgetSettlement`.
