# ADR-002 事业部系统

---

## OAPEFLIR 关联

本文档定义 OAPEFLIR 八阶段认知循环中的以下组件：

- **Observe**：信号采集与统一 DTO
- **Assess**：执行前/后评估与风险判断
- **Plan**：显式规划与 DAG 构建（ADR-060）
- **Execute**：步骤执行与 Dual-Channel 输出
- **Feedback**：信号收集、预处理与 7 类反馈源（ADR-079）
- **Learn**：模式检测与知识提取（ADR-080）
- **Improve**：改进候选评估与 Rollout 状态机（ADR-075）
- **Release**：六级受控发布与自动回滚

## 背景

Automatic Agent 的目标不是只支持编程，而是承载任意可被拆解为工作流的业务。因此业务能力必须以声明式、可插拔、低耦合的方式扩展，而不是写死在平台核心里。

## 决策

将业务能力建模为“事业部”：

- 每个事业部使用一个声明式 YAML 配置描述。
- 配置至少包含 `id`、`name`、`description`、`triggers`、`roles`、`workflow`、`retry`。
- 角色通过 Prompt、模型层级、工具权限、输入输出契约和 preconditions 进行约束。
- 新增一个事业部尽量等价于新增一个配置目录，而不是改动核心代码。

推荐目录形态：

- `divisions/<division>/division.yaml`
- `divisions/<division>/roles/*.prompt.md`
- 可选的 `AGENT.md`、规则文件和事业部私有资源

## 角色模型

角色定义需要显式表达：

- 职责边界：做什么，不做什么。
- 工具能力：最小权限白名单。
- 模型层级：`reasoning`、`coding`、`balanced`、`fast`。
- 输入输出契约：确保步骤之间可以稳定传递数据。
- 并发约束：如 `max_instances`。

角色复用策略：

- 角色定义可以跨事业部复用。
- 同名角色在不同事业部中可以通过工具和边界收缩形成不同能力。

## Workflow 模型

事业部内部 workflow 负责定义：

- 步骤顺序。
- 每个步骤使用的角色。
- 输入字段如何引用上游输出。
- 当前步骤产出的输出键。
- 哪些步骤可以失败重试。

约束：

- workflow 应优先保持线性或轻量 DAG，避免过早变成复杂编排语言。
- 大输出应落到 artifact store，而不是无限制内联在状态表中。

## 动态扩展

HR Agent 负责在现有事业部内动态补角色：

- 分析能力缺口。
- 生成角色契约与 prompt。
- 验证工具、边界和 schema 的一致性。
- 将 workflow 修改作为建议输出，而非自动部署。

边界：

- HR Agent 不自动创建新事业部。
- 新事业部依旧由人工通过 YAML 显式添加。
- 新角色工具集合必须是目标事业部已有工具并集的子集，避免权限膨胀。

## 跨事业部任务

跨事业部任务不是直接绕过事业部系统，而是由 VP 编排：

- 拆分为多个子任务。
- 建立依赖图。
- 确认上游输出可满足下游输入。
- 将各事业部输出聚合为统一结果。

## 结果

优点：

- 业务扩展速度快，符合“平台即公司”的设计目标。
- 新业务天然共享安全、通信、存储和恢复能力。
- 多事业部协作可以统一纳入 VP 编排层处理。

约束：

- 角色工具必须遵守最小权限原则。
- workflow 修改必须经过契约和 schema 兼容性检查。
- 事业部越多，对文档、模板、规则和测试的要求越高。
