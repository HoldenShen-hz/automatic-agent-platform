# ADR-010 商业模型

---

## OAPEFLIR 关联

本文档定义 OAPEFLIR 八阶段认知循环中的以下组件：

- **Observe**：信号采集与统一 DTO
- **Assess**：执行前/后评估与风险判断
- **Plan**：显式规划与 DAG 构建（ADR-060）
- **Execute**：步骤执行与 Dual-Channel 输出
- **Feedback**：信号收集、预处理与 7 类反馈源（ADR-079）
- **Learn**：模式检测与知识提取（ADR-080）
- **Improve**：改进候选评估与 Release 状态机（ADR-075）
- **Release**：六级受控发布与自动回滚

---

- 状态：Accepted
- 决策日期：2026-04-02

## 背景

平台的总体目标包含“盈利良好”，但如果只设计技术架构，不提前定义商业化路径，后续计费、权限、市场和租户能力很难补进来。

## 决策

将商业设计纳入目标架构，但明确实施节奏：

- Phase 1-2：不开收费，以基建成熟和用户反馈为第一目标。
- Phase 3：验证 PMF，启动 Pro 商业化。
- Phase 4：进入 Enterprise、Marketplace、生态和行业解决方案阶段。

## 商业定位

核心定位不是“AI 编程工具”，而是：

- AI 驱动的通用自动化公司运行时。
- 编程只是其中一个事业部。
- 任何可被工作流化的业务都应有机会接入这个平台。

## 商业单位

商业化围绕几类单元展开：

- 事业部：收入单元。
- 用量：计量计费单元。
- Skill / Plugin：生态分发单元。
- 部署模式：社区版、专业版、企业版的差异化单元。

## 定价与节奏

原则上采用三层结构：

- Community：低门槛试用或 BYOK。
- Pro：中小团队和个人创业者。
- Enterprise：私有化、合规、SSO、审计和更强配额控制。

但约束很重要：

- 基建不成熟时不应提前收费。
- 定价数字在市场验证前不应锁死。
- 商业化能力必须和成本模型、租户隔离、安全与用量追踪同步落地。

## 商业化前置能力

至少需要这些技术前提：

- UsageMeter。
- QuotaManager。
- BillingEngine 或等价账单基础设施。
- 多租户隔离。
- 用户体验可观测和错误友好性。
- 合规路线图与企业安全增强预留。

## 结果

优点：

- 商业化不是附加层，而是与成本、租户、安全、渠道同步规划。
- 能更早识别哪些技术能力是商业化前置条件。
- 为 Marketplace、行业解决方案和 Enterprise 能力预留了架构空间。

约束：

- Phase 1-2 不能被商业化需求反向绑架。
- 定价、合规和市场策略都必须在真实市场验证前保留调整空间。
- 任何收费设计都必须与真实成本模型闭环。
