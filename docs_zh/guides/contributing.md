# Contributing

## 目标

拆分后的文档结构用于降低维护难度。今后新增内容应优先写入对应的小文件，而不是继续堆进单一超长文档。

## 文档分工

- `docs_zh/architecture/00-platform-architecture.md`
  平台架构总览，包含目标、总览、核心流程、阶段计划和导航。
- `docs_zh/adr/`
  存放稳定的架构决策，强调背景、决策、结果和交叉引用。
- `docs_zh/guides/`
  存放操作性内容、写作规范和落地流程。
- `docs_zh/`
  不再维护历史 archive 目录，只保留当前仍有效的正式文档。

## 更新规则

- 新的架构决策进入对应 ADR。
- 影响全局认知的变化，同步更新总纲中的摘要和导航。
- 事业部写法、开发流程、维护约束等内容进入 guides。
- 需要追踪来源时，在文件底部补充“来源章节”。
- 发现原始大文档与拆分文档冲突时，以拆分文档为后续维护目标。

## 写作规则

- 一个决策只在一个 ADR 中做主解释，其他文件只链接，不重复长篇复制。
- 尽量使用稳定标题，减少未来链接失效。
- 每次修改至少补一个交叉链接，避免孤立文档。
- 设计、实现顺序、风险和边界应尽量分开表达，避免混成流水账。

## 迁移规则

迁移旧文档内容时：

1. 先判断内容属于总纲、ADR 还是 Guide。
2. 再判断它是“设计决策”还是“操作方法”。
3. 如果只是历史上下文，放摘要和链接，不整段复制。
4. 如果原内容存在重复，以第一次完整出现的正文为准。

## 推荐提交流程

1. 先定位变化属于总纲、ADR 还是 Guide。
2. 修改对应小文件。
3. 检查是否需要补充交叉链接。
4. 若变更影响阶段范围或里程碑，再同步更新总纲。
5. 若只是历史归档变化，不应反向覆盖主文档。
